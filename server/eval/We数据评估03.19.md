# ambient-voice 数据飞轮：架构、评估与分析

## 1. 系统架构

### 1.1 核心理念

用户每天使用 → 自动积累数据 → 蒸馏生成训练集 → 微调端侧模型 → 模型变好 → 用户体验提升 → 继续积累。

## 2. 评估体系

### 2.1 评估总览

三个独立评估，各有不同定位：

```
           训练前                    训练后
             │                        │
    ┌────────┴────────┐      ┌───────┴────────┐
    ▼                 ▼      ▼                ▼
  [A] 数据集质量    [C] 会议模式    [B] 模型效果
  训练数据好不好     产品基线        模型纠错能力
  LLM-as-Judge      端到端 CER+DER   CER+fix/break
```

| 评估 | 问题 | 输入 | 指标 | 用途 |
|------|------|------|------|------|
| A | 训练数据好不好 | merged-pairs.jsonl | LLM Judge 打分 | 决定训练集质量 |
| B | L2 模型纠错能力如何 | voice-history (rawSA) | CER + fix/break 率 | 模型迭代对比 |
| C | WE 会议模式整体表现 | 公开数据集 (AliMeeting/AMI) | CER + DER + RTFx | 产品基线 |

**A 服务于数据飞轮**（训练前过滤低质量数据）
**B 服务于模型迭代**（每次微调后对比 base vs fine-tuned）
**C 服务于产品能力基线**（会议模式在标准数据集上的表现）

评估 B 中 rawSA 就是 SpeechAnalyzer 的实际输出，拿它跑 L2 模型等效于测日常转录的应用效果。

### 2.2 标准工具

| 指标 | 工具 | 版本 | 来源 |
|------|------|------|------|
| CER/WER | jiwer | 4.0.0 | HuggingFace/Whisper 标准 |
| DER | spyder | 0.4.1 | DIHARD/ESPnet 标准 |
| cpWER | meeteval | 0.4.3 | CHiME-7/8 官方 |
| LLM Judge | gemini-3.1-pro | — | 通过 Antigravity Tools 代理 |

## 3. 评估 A：训练数据集质量

### 3.1 方法

用 gemini-3.1-pro 作为 Judge，对 merged-pairs.jsonl 中每条训练对打分。

**Judge 评估维度：**

| 维度 | 定义 | 范围 |
|------|------|------|
| correctness | output 是否正确修正了 input 中的语音识别错误 | 0-1 |
| fidelity | output 是否保持了 input 的原意 | 0-1 |
| naturalness | output 是否自然流畅 | 0-1 |
| suitable | 这条数据是否适合用于训练 | true/false |

**不适合训练的判定标准：**
- output 被截断不完整
- output 比 input 更差
- 过度改写（超出 ASR 纠错范围）
- 脏数据（特殊符号等）

### 3.2 参数

```
脚本：server/eval/scripts/judge_data_quality.py
Judge：gemini-3.1-pro @ http://127.0.0.1:8045
数据：merged-pairs.jsonl (122 条)
来源分布：human=37, whisper=48, gemini=37
```

### 3.3 结果

| 来源 | 条数 | 合格 | 合格率 | correctness | fidelity | naturalness |
|------|------|------|--------|-------------|----------|-------------|
| gemini | 37 | 22 | **59.5%** | 0.66 | 0.87 | 0.73 |
| whisper | 48 | 7 | **14.6%** | 0.43 | 0.58 | 0.53 |
| human | 37 | 1 | **2.7%** | 0.17 | 0.19 | 0.33 |
| **总计** | **122** | **30** | **24.6%** | **0.42** | **0.55** | **0.53** |

冲突对：44 条（36%）

### 3.4 分析

**1. Human corrections 质量最差（2.7% 合格率）**

这违反直觉——人工纠正应该是最高质量的。36 条不合格中，Judge 标记的问题分布：
- **截断**（21 条）：output 在中间断开，丢失后半段内容。CorrectionCapture 通过 AX API 读取焦点元素文本时只读到部分内容
- **脏数据**（18 条，与截断有重叠）：output 中混入终端 prompt 符号 `❯`，说明 AX API 读取时把非用户输入的内容也采集了
- **过度改写**（6 条）：用户修改幅度过大，从口语重写为书面语（如"试一下"→"试试"），超出 ASR 纠错范围

Judge 的典型反馈：
> "output被严重截断不完整（句末'方'字未成词），且完全丢失了input后半部分大量关键信息"
> "output开头包含异常特殊符号'❯'"

**结论：CorrectionCapture 存在两个采集缺陷——截断和脏数据混入。这是当前数据质量的最大瓶颈，但属于工程 bug 而非方案问题。修复后 human corrections 应成为最高质量来源。**

**2. Gemini 来源最好（59.5% 合格率）**

Gemini Flash 作为蒸馏教师，fidelity 0.87 最高——它最尊重原意，只做必要修改。但仍有 40% 不合格（15 条），Judge 标记的主要问题：
- **output 截断**（6 条）：句子在中间断开，如"我试一下能不能听到我的"缺失"声音"。部分发生在短句上（原文仅 20 字），排除了 max_tokens=256 的限制，可能是 Gemini 自身生成行为提前终止
- **未纠错**（5 条）：input 和 output 完全一致，但原文有明显错误（如"细细看"应为"试试看"），模型未识别出错误
- **丢失标点**（2 条）：output 反而比 input 少了句末标点
- **过度删减**（2 条）：去除口水词时连带删除了有效内容

**3. Whisper 来源居中（14.6% 合格率）**

Whisper 的问题是它做的是**重新转写而非纠正**——它从音频独立产出文本，和原始 rawSA 的对齐度低。Judge 认为很多 Whisper 输出虽然可能更准确，但和 input 的对应关系不明确，不适合作为 "input→output" 训练对。

**4. 高冲突率（36%）说明双路蒸馏一致性低**

同一条 input，Whisper 和 Gemini 给出不同的 output，且两者质量都不高。在数据量少时，冲突对的训练价值存疑。

## 4. 评估 B：L2 润色模型效果

### 4.1 方法

用 voice-history.jsonl 作为测试集（105 条真实使用数据），将 rawSA 送入不同 L2 模型润色，对比润色后的 CER。

**reference 来源：** voice-history 中的 finalText（经过 L1+L2 后实际注入用户的文本）

**分类标准：**
- fix：模型润色后 CER 比原始低（改对了）
- break：模型润色后 CER 比原始高（改错了）
- identity：模型没改动

### 4.2 参数

```
脚本：server/eval/scripts/eval_l2_model.py
测试集：~/.we/voice-history.jsonl (105 条)
CER 工具：jiwer
模型 A：qwen3.5:4b @ ollama (100.64.0.3:11434)
模型 B：gemini-3-flash @ Antigravity Tools (127.0.0.1:8045)
```

### 4.3 结果

| 模型 | N | CER_raw | CER_polished | ∆CER | Fix% | Break% | 延迟 |
|------|---|---------|-------------|------|------|--------|------|
| qwen3.5:4b | 104 | 71.4% | **11.6%** | **+59.8** | **85.6%** | **6.7%** | 3.0s |
| gemini-3-flash | 104 | 71.1% | 208.3% | -137.3 | 57.7% | 26.0% | 6.2s |

### 4.4 分析

**1. qwen3.5:4b 全面优于 gemini-3-flash**

- CER 从 71.4% 降到 11.6%（降幅 59.8 个百分点）
- 85.6% 的样本被改对，只有 6.7% 被改错
- 延迟 3 秒，用户体感可接受

**2. gemini-3-flash 严重过度改写**

CER 208.3% 意味着模型输出的文本比原始 rawSA 偏离参考答案更远。原因：
- Gemini Flash 倾向于重写而非纠错（添加大量原文没有的内容）
- 它不是为短文本 ASR 后处理设计的，而是通用对话模型
- 26% 的 Break 率说明超过四分之一的样本被改得更差

**3. qwen3.5:4b 作为端侧 L2 的定位是合理的**

小模型在纠错任务上的优势：
- 参数少 → 改动保守 → Break 率低
- 本地推理 → 延迟低
- 经过 system prompt 约束 → "只输出结果，不解释不扩写"

**4. CER_raw 71.4% 的含义**

这个数字看起来很高，但 voice-history 中很多是早期测试数据（"喂喂喂有没有录音"等），rawSA 和 finalText 差异大。随着实际使用数据增多，这个基线会更有参考意义。

**5. 微调的预期目标**

当前 base model (qwen3.5:4b) 已经达到 CER 11.6%。微调的目标是：
- 进一步降低 CER（目标 <8%）
- 降低 Break 率（目标 <3%）
- 提升领域词汇识别（如 "蒸馏"、"Claude Code"、"MCP" 等当前识别不好的专有名词）

## 5. 评估 C：WE 会议模式端到端

### 5.1 方法

通过 WE 的 `MeetingSession.runFromFile()` 完整链路处理公开数据集音频，评估转写质量（CER）和说话人分离质量（DER）。

**完整链路：**
```
WAV 文件 → SpeechAnalyzer(inputAudioFile) → 转写段(含 volatile + audioTimeRange)
         → FluidAudio performCompleteDiarization() → 说话人分离
         → alignTranscriptionWithDiarization() → 对齐
         → MeetingExporter → Markdown
```

### 5.5 分析

**1. 远场 vs 近场 CER 差 15 个百分点**

远场 40.0% → 近场 25%（去异常值）。原因明确：远场取的是 8 通道阵列的单通道（ch0），无 beamforming，信噪比低。近场是头戴麦直接采集。这不是产品的目标使用场景——用户实际用蓝牙耳机或近场麦克风。

**2. 说话人检测系统性欠分割**

4 人会议普遍只检测到 1-3 人。FluidAudio 在远场高重叠（>30%）条件下的聚类表现较差。2 人会议（R8009 系列）检测基本准确。

**3. DER 中 Confusion 是主要错误**

AliMeeting 整体 DER 48.5% 中，Confusion（说话人标错）占 24.1%，是最大的错误来源。Miss（漏检）21.6%，FA（误检）只有 2.7%。系统不会无中生有，但会把 A 说的话标成 B。

**4. 处理速度充裕**

RTFx 74-131x，30 分钟会议 15-25 秒处理完。内存峰值 <1GB，8GB 设备无压力。速度不是瓶颈。

**5. AMI vs AliMeeting DER 差异大（23% vs 49%）**

主要因素：AliMeeting 重叠率 >30%（AMI ~10-20%），且 AliMeeting 是远场单通道。这两个数据集的比较说明**重叠率是 DER 的最大变量**。

## 6. 当前状态与下一步

### 6.1 已完成

- [x] 数据飞轮全链路自动化（采集 → 蒸馏 → 同步 → 合并）
- [x] 评估 A：数据集质量评估（122 条，gemini-3.1-pro Judge）
- [x] 评估 B：模型效果评估（qwen3.5:4b vs gemini-3-flash，105 条）
- [x] 评估 C：会议模式端到端评估（AliMeeting 远/近场 + AMI）
- [x] 服务器数据目录规范化（data/<user>/ 多用户结构）
- [x] 评估工具链（jiwer + spyder + meeteval + LLM Judge）

### 6.2 关键发现

1. **训练数据质量是当前最大瓶颈**：122 条中只有 30 条合格（24.6%）
2. **CorrectionCapture 截断 bug** 导致 human corrections 几乎全废
3. **qwen3.5:4b 作为端侧 L2 已经有效**：CER 71% → 12%，Fix 86%
4. **Gemini Flash 不适合做端侧 ASR 后处理**：过度改写，Break 率 26%
5. **会议模式瓶颈在说话人分离**而非转写，尤其是高重叠场景

### 6.3 下一步

**优先级 1：提升训练数据质量**
- 修复 CorrectionCapture 截断 bug
- 用 gemini-3.1-pro 对 rawSA 直接生成高质量纠正（替代现有低质量路线）
- 用 Judge 过滤后重新构建训练集

**优先级 2：执行微调**
- 用清洗后的高质量数据跑 train_qlora.py
- 评估 B 对比：base model vs fine-tuned
- 目标：CER <8%，Break <3%

**优先级 3：持续积累**
- 日常使用积累更多真实数据
- 修复 CorrectionCapture 后积累高质量 human corrections
- 每 100 条新数据一轮迭代
